# Deployment Strategy (Artifact Publish)

成果物をレジストリへ公開する戦略。反映先は稼働環境ではなくレジストリで、consumer が取り込むまで反映されない。

## 概要

共有ライブラリやパッケージ、公開イメージなど、他リポ / 他者が取り込む配布物を扱うリポで採用する。タグ push を起点に、レジストリ (npm / NuGet / PyPI / Go module / コンテナレジストリ等) へ publish する。「デプロイ」ではなく「配布物の公開」であり、後方互換の境界が consumer 側に効くため SemVer が特に重要になる。

## 環境とトリガー

| 公開先 | トリガー |
|---|---|
| レジストリ | タグ push → publish |

モノレポで複数パッケージを持つ場合は、パッケージ単位のタグ接頭辞を付ける (例: `packages/<name>/vX.Y.Z`)。どのパッケージを bump するかは前回タグとの差分で判定する。

## ゲート

タグを打つこと (＝公開バージョンの確定) がゲート。

## 反映方式

publish。CI がレジストリへ成果物を公開する。consumer 側がバージョンを更新して取り込むまで、consumer の動作には反映されない。

## タグ・バージョニング

SemVer を厳守する (後方互換の境界が consumer に直接効くため)。

## 想定リポ / 選定の目安

- 共有ライブラリ、API 契約パッケージ、公開コンテナイメージ
