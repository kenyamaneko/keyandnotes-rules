# Deployment Strategy (Merge → Dev, Tag → Prod)

dev と prod の2環境を持ち、マージで dev、タグで prod へ自動デプロイする戦略。

## 概要

`main` へのマージで dev に自動デプロイして継続的に統合検証し、検証が済んだらタグを打って同一コミットを prod に自動デプロイする。コミット単位で環境デプロイ先を切り替えられるリポ (コンテナ / PaaS 上のアプリ等) で採用する。

## 環境とトリガー

| 環境 | トリガー |
|---|---|
| dev | `main` へのマージ → 自動デプロイ |
| prod | タグ (`vX.Y.Z`) push → 自動デプロイ |

## ゲート

タグを打つことが手動ゲート。dev で検証したうえで、人がタグを打って prod へ昇格させる。

## 反映方式

CI がマージ / タグを起点に成果物をビルドし、対象環境へデプロイする。dev で検証した成果物をそのまま prod へ昇格させるため、タグ時に再ビルドせず同一成果物を用いることが望ましい。

## タグ・バージョニング

SemVer 形式のタグ (`vX.Y.Z`) を人が手動で打つ。

## ロールアウト

ローリング更新。詳細はデプロイ先のサービスに従う。

## 想定リポ / 選定の目安

- コンテナ / PaaS 上で稼働し、コミット単位に環境を切り替えられるアプリ
- stg を挟む必要があるものは `gitops-sync.md` / `manual-pipeline.md` を検討する
