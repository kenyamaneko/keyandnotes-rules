## [base] バージョニング

バージョン番号 (`X.Y.Z`) の上げ方は、そのリポに直接のユーザー (エンドユーザー) がいるかで分かれる。どちらも上から順に判定し、最初に当てはまったものを採る。

### 直接のユーザーがいる場合 (アプリ・エンドユーザー向けサービスなど)

ユーザーから見た変化の大きさで決める。

- 機能が増えていない → **PATCH**
  - バグ修正、パフォーマンス改善、内部リファクタ、ドキュメント・文言修正など
- 見える機能追加・改善だが、体験を一新するほどではない → **MINOR**
  - 既存機能へのオプション追加、小さな新機能、UI 改善など
- ユーザー体験が大きく変わる規模の機能追加 → **MAJOR**
  - 新しい主要機能・画面・モードの追加、既存の使い方の作り直しなど。目安は年1回程度

### 直接のユーザーを想定しない場合 (ライブラリ・パッケージ・API 契約など、他のコードから取り込まれるもの)

取り込む側の後方互換で決める。

- 後方互換を壊さない修正 → **PATCH**
  - バグ修正、内部リファクタ、ドキュメント修正など
- 後方互換を保った機能追加 → **MINOR**
- 後方互換を壊す変更 → **MAJOR**
  - API・スキーマ・シグネチャの破壊的変更など、取り込む側の修正が必要になるもの

上位の桁を上げたら下位の桁は 0 に戻す (例: `1.4.2` から MINOR なら `1.5.0`、MAJOR なら `2.0.0`)。迷ったら小さいほうに倒す。

## [base] CI方針

- コスト削減のため、新規 GitHub Actions workflow を追加するときは `paths-ignore` / `timeout-minutes` / `concurrency` を必ず設定する
- 可読性を高めるため、ジョブの各ステップには `name` を付与する

## [base] 指示遵守

- ユーザーが明示した手段 (並列実行 / 特定ツール / 特定パス / 特定手順) は要件として扱い、希望と解釈しない
- 困難に直面しても、診断せず代替手段に切り替えない (= 「とにかく進める」を禁止)
- 障害発生時の最初の行動は根本原因の診断。迂回判断は原因を共有してから相談する

## [base] 判断権はユーザに属する

判断はユーザが持つ。Claude は実行者であって判断者ではない。

- 「Claude が判断してユーザが楽になる」と感じた瞬間が最も危険な合図。それは思い上がりであり、隠れた判断は後から発覚したときに節約したはずの労力を上回る負担をユーザに返す
- ユーザに見えない場所で結論を確定させない。判断材料を出し、判定はユーザに残す
- ルール・指示・仕様を softening / 簡略化 / 例外扱いすることは全て判断行為。揺らいだら必ずユーザに上げる
- audit / 報告 / 列挙で合否・軽重・優先度を Claude が pre-categorize しない

## [base] ルール違反への対応

- ルール違反がある状態でコミットしない。検出された違反は Claude が自律的に解消してからコミットする
- 違反を許容してよいかの判断をユーザに求めない。正当な逸脱理由がない単なる違反は、問わずに修正する
- ルールから外れる方が良いと判断した場合に限り、その理由を明記した上でユーザに判断を仰ぐ

## [base] 禁止事項

- Google Cloud のことを GCP と表現してはいけない（GCP は旧名称のため）
- 生成済みコードを手で書き換えない。型・スキーマ・定数等は SSoT を変更してから生成スクリプトを再実行する（該当する SSoT がある場合に適用）
- 各リポルートの `CLAUDE.md` を Claude が書き換えない (common への動線指示のみで、運用上 stable に保つ必要があるため)
- `rules/` 配下のファイルを Claude が書き換えない (人間が運用するため、提案までに留める)
- 保守性を高めるため、原則としてエラー回避のためのワークアラウンドを書かない
