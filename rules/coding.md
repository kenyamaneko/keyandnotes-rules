# コーディング規約

Key and Notes 各プロジェクト共通のコーディング規約。コードを書くときに従う。

## 設計思想

- エラーは握りつぶさず根本解決する。ログだけして握りつぶすのは禁止
- デフォルト値へのフォールバックを行わない。意図しない値になるならエラーにする
- 一つの関数に複数の責務を負わせない

## コーディング方針

- マジックナンバーを使用しない。定数化し、数値の用途がわかるようにする
- コメントは原則書かない。書いてよいのは、公開関数・クラス・型に付与する docs コメントと、実装から読み取れない意図を示す Why コメントの二種類のみ
- モジュール外部から参照される (公開の) 関数・クラス・型には docs コメントを付与する。ファイル内でしか使わない内部専用の関数には付与しない
  - 要約のみを書く。実装手順や設計判断の列挙はしない
  - 設計レベルの Why は ARCHITECTURE.md に、行レベルの Why はその行のインラインコメントに分離する
  - 実装の言葉ではなく仕様の言葉で書く
- 実装との二重管理になるので What コメントを書いてはいけない
  - Why がかかる先を示すために、処理内容を要約の粒度で添えるのは可
  - 機能や画面要素を削除した後に「〜はしない」「〜は表示しない」と否定形の説明を残さない。動作しないことは現在のコードから読み取れるので、Why のない否定の記述はノイズになる。ドキュメントでも同じで、あえて対象外にしたことを書くなら理由 (why) を添える
- 実装から読み取れない意図は Why コメントとして記述する。設計レベルの意図なら ARCHITECTURE.md に記述する
  - 変更の歴史的経緯（以前はこうだった／いつ何をなぜ変えた）は Why ではない。現在のコードを読むのに不要なので、コメントに書かず、残すなら ADR や git 履歴に置く
  - 特に ADR 番号の参照（「ADR-xxx に基づく」等）をコメントに書かない。ADR の内容が Why なら本文に直接書く。番号参照は経緯であって Why ではなく、ADR の改番・統廃合で陳腐化する
- アプリ間の契約 (エンドポイント・イベント名・ヘッダ名) はリテラルで書かず、所有サービスが発行する API 契約パッケージを参照する
- 未実装の TODO をコミットに残さない。残す場合は Issue を立ててリンクする
- 関数名は動詞から始め、何をする関数かを明確にすること
- 今使わないコードは残さない (動線があってもロジック的に呼ばれないコードを含む)
- 実装を変更したら関連 docs を同期する (ただし過去の ADR は変更しない)

## 命名規約

- 関数・メソッドは名前から振る舞いが読み取れるようにする (原則として動詞で始める)
- 真偽を表す関数・変数は、真偽が読み取れる接頭辞を付ける (is / has / can / should 等)
- 一般的でない略語を使わない (ドメインで定着した語・各言語の確立した短縮語は可)
- 上記の具体化と言語固有の命名慣用は rules/lang/<lang>.md に従う。慣用が本節と衝突する場合は lang を優先する

## ログ方針

- 構造化されたログを出力する
- ログ出力すべき事象
  - 呼び出し元に伝搬させられないエラーの補足
  - エラーではないが、その場所でしか気づけない事象 (未対応のイベント種別など)
- ログレベル
  - `Info`: 起動・停止・正常系の事象
  - `Warn`: システムの運用に影響を与えない、あるいは影響が軽微な事象
  - `Error`: システムの運用に支障をきたす事象
