# コーディング規約

Key and Notes 各プロジェクト共通のコーディング規約。コードを書くときに従う。

## 設計思想

- エラーは握りつぶさず根本解決する。ログだけして握りつぶすのは禁止
- デフォルト値へのフォールバックを行わない。意図しない値になるならエラーにする
- 一つの関数に複数の責務を負わせない
- 新しいコードが既存コードと同じ概念を表し、片方の変更がもう片方にも及ぶ関係にあるなら、部品として切り出して共通化する
- 共通化に既存コードの修正が要る場合も、修正を避けない。既存コードに触れないことを理由に、共通化すべき部分を重複させたまま実装しない
- 業務ロジックは DB・外部 API・フレームワーク・乱数・時刻など、外部の具体的な実装に直接依存しない。ロジック側でインターフェースを定義し、具体的な実装はそのインターフェースを実装する側に置く。
  - 外部の変更がロジックに波及せず、実装を差し替えられるようになる
  - テストで本物と偽物を差し替えられ、テストのしやすさが向上する

## コーディング方針

- マジックナンバーを使用しない。定数化し、数値の用途がわかるようにする
- コメントは原則書かない。書いてよいのは、関数・クラス・型に付与する docs コメントと、実装から読み取れない意図を示す Why コメントの二種類のみ
- モジュール外部から参照される (公開の) 関数・クラス・型には docs コメントを付与する。非公開の関数・クラス・型に付与するかは任意
  - 要約のみを書く。実装手順や設計判断の列挙はしない
  - 設計上のトレードオフによる Why は ADR に、仕様・業務ルールとしての Why は BDR に、行レベルの Why はその行のインラインコメントに分離する
  - 実装の言葉ではなく仕様の言葉で書く
- 実装との二重管理になるので What コメントを書いてはいけない
  - Why がかかる先を示すために、処理内容を要約の粒度で添えるのは可
  - 機能や画面要素を削除した後に「〜はしない」「〜は表示しない」と否定形の説明を残さない。動作しないことは現在のコードから読み取れるので、Why のない否定の記述はノイズになる。ドキュメントでも同じで、あえて対象外にしたことを書くなら理由 (why) を添える
- 実装から読み取れない意図は Why コメントとして記述する。設計上のトレードオフなら ADR に、仕様・業務ルールとしての判断なら BDR に記述する
  - 変更の歴史的経緯（以前はこうだった／いつ何をなぜ変えた）は Why ではない。現在のコードを読むのに不要なので、コメントに書かず、残すなら ADR・BDR や git 履歴に置く
  - 特に ADR・BDR 番号の参照（「ADR-xxx に基づく」等）をコメントに書かない。ADR・BDR の内容が Why なら本文に直接書く
  - Why コメントは「〜のため、〜する」の一文で書く。理由を一つに絞れない、または代替案の比較や却下理由に触れる場合は行レベルの Why ではなく設計上のトレードオフであり、ADR に書く
- アプリ間の契約 (エンドポイント・イベント名・ヘッダ名) はリテラルで書かず、所有サービスが発行する API 契約パッケージを参照する
- 未実装の TODO をコミットに残さない。残す場合は Issue を立ててリンクする
- 関数名は動詞から始め、何をする関数かを明確にすること
- 今使わないコードは残さない (動線があってもロジック的に呼ばれないコードを含む)
- 実装を変更したら関連 docs を同期する (ただし過去の ADR・BDR は変更しない)

## 命名規約

- 関数・メソッドは名前から振る舞いが読み取れるようにする (原則として動詞で始める)
- 真偽を表す関数・変数は、真偽が読み取れる接頭辞を付ける (is / has / can / should 等)
- 一般的でない略語を使わない (ドメインで定着した語・各言語の確立した短縮語は可)
- 同じ概念には常に同じ語を使う。新しい識別子・ラベルを付ける前に、仕様書・既存コード・ドキュメントで同じ概念を指す語が既に使われていないか確認し、あればそれに揃える。表記が割れている、またはどの語を正とするか決められない場合は、Claude が語を選ばずユーザーに確認する。
- 上記の具体化と言語固有の命名慣用は rules/lang/<lang>.md に従う。慣用が本節と衝突する場合は lang を優先する
- テスト名の詳細な指針は testing.md に従う

## ログ方針

- 構造化されたログを出力する
- ログ出力すべき事象
  - 呼び出し元に伝搬させられないエラーの補足
  - エラーではないが、その場所でしか気づけない事象 (未対応のイベント種別など)
- ログレベル
  - `Info`: 起動・停止・正常系の事象
  - `Warn`: システムの運用に影響を与えない、あるいは影響が軽微な事象
  - `Error`: システムの運用に支障をきたす事象
