# テストの書き方

Key and Notes 各プロジェクト共通のテスト規約。`principles.md` から参照される。
言語ごとの書き方 (テーブル駆動・ネストクラス等) は `rules/lang/<lang>.md` を見る。

## 基本方針

テストは実装をなぞるのではなく仕様を確かめる。

## 単体テストと結合テストの切り分け

- 関数に固有のロジック (計算式・分類・変換のルールなど) があるなら、期待値を具体値で書く単体テストを作成する。
- ロジックを持たない関数に対しては単体テストを行わず、固有のロジックを持つ関数まで呼び出し元を遡り、結合テストの一環としてカバレッジを通せれば良い。

## 大原則

- 失敗した時の処理が正しいかを確認するため、正常系だけでなく異常系のケースを作成する。
- リファクタしても壊れないテストにするため、外から観測できる結果だけをアサーションの対象にする。
- 人間がテストの網羅性を確認しやすいように、テスト名と中身を一致させる。テスト名と説明は仕様の言葉で書く。
- 既存テストを別の形 (公開経路の検証など) に作り直すときは、置き換え先のテストが通ることを確認してから元のテストを消す。
  - なぜ: 先に消すと、置き換えが完成するまでその仕様が無検証になり、穴が残っても気づけないため。

## テストの命名

テスト名から仕様書を生成するため、テスト名だけを読めば仕様がわかるように命名する。

テストには名前を付ける場所が二つあり、それぞれに書く内容を分ける。

- **テスト全体の名前**：テスト対象の要素を書く (送料計算・会員ランク判定・日付変換などのロジック名)。
- **各ケースの名前**：そのケースのシナリオを「〜のとき、〜すると、〜になる」の一文で書く。この三つの部分を順に **Given** (前提)・**When** (操作)・**Then** (期待する結果) と呼ぶ。操作 (When) がテスト全体の名前から自明なら省略してよい (例の「注文金額が 3000 円のとき、送料は無料になる」は When を省いた形)。

例：送料計算のテストなら、テスト全体の名前は「送料計算」、ケース名は「注文金額が 3000 円のとき、送料は無料になる」のように書く。

- どの構文が「テスト全体の名前」と「各ケースの名前」に当たるかは言語で異なる。rules/lang/<lang>.md に従う。
- 同じ前提のケースが多いときは、Given を中間のグルーピング (ネストクラスや describe の入れ子) に括り出してよい。
- テーブル駆動 (データ駆動) では When がテーブル全体で共有されるので、各ケース名は Given と Then だけにする。
- 境界値の網羅を名前の一覧だけで確認できるようにするため、ケース名には具体値を含める。
  - ただし、具体値がテスト専用のダミーで、検証対象が値そのものでなく境界の関係なら、「下限のとき」「上限のとき」のような相対表現でよい。
- ケース名・DisplayName など自由文で書ける箇所は日本語の仕様の言葉で書く。
  - ただし、キー名・列挙値・特殊値 (NaN 等) など、それ自体が仕様に現れる (エラーメッセージや保存形式に出る) 識別子は、日本語に言い換えずそのまま使う。

## ロジックのテスト

- バグは境界に出やすいため、境界値を確かめる。条件の境界をまたぐ両側の値を入れる (例: 整数で a>0 なら 0 と 1)。
- 件数や有無で扱いが変わるなら、0 件・1 件・複数件、「無い」と「空」をケースに分ける。
- テストコードに if 文を書かず、条件ごとに独立したテストケースとして並べる。
  - 前提 (Given) にifを使うと、どんなケースが実際に走ったか見えにくく、網羅できているか分かりづらくなる
  - アサーションにifを使うと、本来のアサーションの前に別のアサーションが紛れ込む。1つのテストは1つのアサーションだけを確かめるべきだが、if はこれを壊す。
- 実データの仕様変更でテストが壊れるのを防ぐため、テストでは本番の実データ ID をそのまま使わず、テスト専用のダミー値を用いる。

## 結合テスト
- 結合を確かめるテストは、途中の状態を手で用意して割り込ませず、公開された入口 (API・画面操作など)から実行する。
  - なぜ: 途中から始めると、入口から結果までの経路が壊れていても気づけないため。
- 操作した結果まで確かめる。「例外が出ない」「エラーが無い」「件数が合う」だけで終えず、操作の後に実際にどうなったか (保存された内容・返ってきた値・変化した状態) まで踏み込む。
- エミュレータや DB コンテナなど前提のインフラが使えない環境では、テストを沈黙スキップせず失敗させる。
  - なぜ: スキップが常態化すると、検証されていないのに全体が通ったように見えるため。

## GUI (画面) のテスト

- 画面に見える結果で判断する。画面の切り替わり・表示内容・エラー表示を確かめ、モックがどう呼ばれたか・内部の値は検証しない。
- モックにするのは外部との境界だけ。通信・WebSocket・ログインなどを代用に差し替え、画面を組み立てる内部の部品は本物を使う。
  - なぜ: 部品どうしの繋がりが壊れていても気づけるようにするため。
- API 通信のモックは MSW で HTTP 境界 (ネットワーク層) をモックする。api クライアント / モジュールを `vi.mock` 等で直接差し替えない。
  - なぜ: api モジュールを差し替えるとリクエスト組み立て〜レスポンス処理の経路が飛ばされる。ネットワーク層でモックすれば実際の通信経路を通して検証できるため。
- 見た目や実装の詳細に依存させない。CSS クラス名・アニメーションの文字列・配置の係数・固定の待ち時間などを確かめの対象にしない。
  - なぜ: 見た目を変えただけで壊れ、ユーザーに見える動きを確かめていないため。状態を見分けたいときは意味のある属性 (role / aria-* / data-state) で確かめる。
- フロントエンドは React を前提とする。コンポーネントは Testing Library (`@testing-library/react` + `@testing-library/user-event`) を用い、操作は実際のユーザー操作として発火させる。

## アンチパターン

- 入力と出力の対応を、変換関数を直接呼んで突き合わせてはいけない。
  - 例: エラー→HTTP ステータスの変換関数を単体で呼び、対応を 1 件ずつ確かめる。コードの分岐を書き写しただけで、その状況で本当にその出力が返るかは確かめていない。
  - 除外: 契約の SSoT が、実際の通信を介してこの変換を確かめている場合。
- 生成物の文字列を期待文字列とそのまま突き合わせてはいけない。
  - 例: 生成した SQL やコードを文字列比較で確かめる。書式を変えただけで壊れやすく、実行した結果が正しいかは確かめていない。
  - 完全一致を部分一致に緩めても、実行した結果を確かめていない点は変わらない。生成物を実行し、その結果 (更新された行・出力された内容) を確かめる。
- 依存を欠けた状態で渡し、早期エラーだけを確かめてはいけない。
  - 例: ハンドラに依存を欠けた状態(nil/null/未設定など) で渡し、入力エラーの 400 や「落ちない」ことだけを確かめる。本来の処理(依存を呼んで結果を返す) に届かず、配線の欠落を見逃す。
- モックが「呼ばれたこと」「呼ばれた回数」だけを確かめて終えてはいけない。
  - 例: 配信関数が 1 回呼ばれたことだけを確かめる。誤った内容が渡っていても緑のままになる。境界のモックで捕捉した引数の内容や、操作の後に実際に起きた結果 (保存・配信された内容) まで確かめる。
- 不具合修正のあとに追加するテストは、直した不具合の再発だけをピンポイントで確かめてはいけない。
  - 例: 「送料が負の値にならない」のように、修正前に起きていた症状が出ないことだけを確かめる。同じ処理に潜む別の不具合には反応せず、本来の仕様に沿った期待値 (例: 「送料は 500 円になる」) を確かめれば、直したバグも含めて仕様から外れる結果を検出できる。
- 期待値をアサートした上に、その裏返しの「起こらないこと」を重ねて確かめてはいけない。
  - 例: 送料が 500 円であることを確かめた上で、さらに送料が -1 円でないことも確かめる。期待値のアサーションが通れば否定は自動的に成り立ち、何も確かめを足していない。
- 実装をテストに書き写してはいけない。
  - 例: フィールドのコピーや定数をそのまま写す、期待値を
  テスト対象と同じ計算で作る。実装とテストが二重管理になり、両方に同じバグが入れば気づけない。
  - 本番コードの定数や関数を取り込んで期待値を組み立てるのも同じ書き写しであり、定義が誤っていても両側が同時に変わり検出できない。期待値はテスト側のリテラルで書く。
  - 除外: 純粋な関数 (計算式・分類・命名規則) を具体値で直接確かめる場合。
- 本番に無い検証ロジックをテスト内で作って確かめてはいけない。
  - 例: 本番に存在しないロジックや変換をテスト内に書き、自前の入力に当てる。テスト内で完結し、本番コードを何も通っていない。
- テスト対象でないライブラリそのものの挙動を確かめてはいけない。ライブラリの品質はライブラリ開発者の責任であり、このプロジェクトで検証しても仕方がない。
- テスト名に実装の言葉 (変数名・関数名など) を含めてはいけない。
  - 例: ケース名に `shippingFee` や `calcShippingFee()` のような変数名・関数名をそのまま使ってはいけない。仕様の言葉に言い換えられておらず、実装を読まないと意味がわからない。
- テストケースの意図やシナリオをコメントで説明してはいけない。
  - 例: 「二つの集計が別々に反映されることを、値が食い違う入力で確かめる」とコメントに書く代わりに、その内容をケース名に書く。テスト名だけで仕様がわかる方針に反し、コメントを消すと情報が失われる。
