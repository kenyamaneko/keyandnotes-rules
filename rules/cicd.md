# バージョニングとCI方針

Key and Notes 各プロジェクト共通のバージョニング・CI規約。`principles.md` から参照される。

## バージョニング

バージョン番号 (`X.Y.Z`) の上げ方は、そのリポに直接のユーザー (エンドユーザー) がいるかで分かれる。どちらも上から順に判定し、最初に当てはまったものを採る。

### 直接のユーザーがいる場合 (アプリ・エンドユーザー向けサービスなど)

ユーザーから見た変化の大きさで決める。

- 機能が増えていない → **PATCH**
  - バグ修正、パフォーマンス改善、内部リファクタ、ドキュメント・文言修正など
- 見える機能追加・改善だが、体験を一新するほどではない → **MINOR**
  - 既存機能へのオプション追加、小さな新機能、UI 改善など
- ユーザー体験が大きく変わる規模の機能追加 → **MAJOR**
  - 新しい主要機能・画面・モードの追加、既存の使い方の作り直しなど。目安は年1回程度

### 直接のユーザーを想定しない場合 (ライブラリ・パッケージ・API 契約など、他のコードから取り込まれるもの)

取り込む側の後方互換で決める。

- 後方互換を壊さない修正 → **PATCH**
  - バグ修正、内部リファクタ、ドキュメント修正など
- 後方互換を保った機能追加 → **MINOR**
- 後方互換を壊す変更 → **MAJOR**
  - API・スキーマ・シグネチャの破壊的変更など、取り込む側の修正が必要になるもの

上位の桁を上げたら下位の桁は 0 に戻す (例: `1.4.2` から MINOR なら `1.5.0`、MAJOR なら `2.0.0`)。迷ったら小さいほうに倒す。

## CI方針

- コスト削減のため、新規 GitHub Actions workflow には timeout-minutes / concurrency を必ず設定する
- コスト削減と CI パイプライン実行時間短縮のため push / pull_request トリガーの workflow は、ドキュメントのみの変更ではテスト・ビルドを実行しない。
  - required status check を使わない workflow の場合
    - paths-ignore を設定し、ドキュメント修正のみの PR をテストなしでマージできる
  - required status check を使う workflow の場合
    - paths-ignore は使わない (トリガー自体が止まり、check 未報告のまま PR がマージ不可になるため)。
    - 代わりに変更判定ジョブを置き、パイプライン実行不要の差分なら後続 job を needs + if でスキップする。スキップは required check の合格扱いになるため、ドキュメント修正のみの PR をテストなしでマージできる
- workflow_dispatch / schedule / workflow_call のみの workflow に paths フィルタは適用しない (変更ファイルの概念がないため)
- 可読性を高めるため、ジョブの各ステップには name を付与する
