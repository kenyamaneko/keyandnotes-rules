## [lang/go] 設計思想

- 保守性と拡張性を最大限に高めるため、クリーンアーキテクチャを遵守する
  - ビジネスロジックは外部アダプターに依存しない
  - 純粋な port や delivery 層とロジックを分離する
  - リポジトリ層は純粋な外部永続装置へのアクセスであり、ロジックを持たない
  - リポジトリ層は外部永続装置以外に依存しないため、API 契約の型を使わない

## [lang/go] テスト方針

- テストは標準 `testing` と `stretchr/testify` を用いる
- テストコードはテーブル駆動 (`t.Run` でケース分岐) で書く
  - 単一シナリオや前提条件が異なるテストは無理にテーブルにしない(1 行だけのテーブルを作らない)
- テストの命名は testing.md「テストの命名」を次のとおり割り当てる
  - テスト関数名 = `Test<対象>` (例: `TestShippingFee`)。Go がテスト関数と認識するのは `Test` で始まる関数名だけで、日本語にできない。仕様は日本語で書くため、テスト関数名は仕様ドキュメントに出さない
  - テスト対象の要素 = 関数の直下に置く `t.Run` の名前に日本語で書く (例: `t.Run("送料計算", ...)`)
  - 各ケースの名前 = その `t.Run` の中に置く、ケースごとの `t.Run` の名前 (テーブル駆動ならテーブルの name フィールド) に、Given + Then を日本語で書く (例: `"注文金額が3000円のとき、送料は無料になる"`)
  - ケースが 1 つだけのときも、テーブルにはせずその 1 ケースを `t.Run` で書く
    - なぜ: 仕様ドキュメントに全テストがケース名の粒度で出力されるようにするため

## [lang/go] コード生成

- 型コードは REST 契約 `data/openapi.yaml`・Pub/Sub 契約 `data/asyncapi.yaml` を SSoT とし、`make generate-types` (`scripts/generate_types.sh`) で再生成する

## [lang/go] docs コメント

- 型・関数・メソッドには godoc コメントを書く (識別子名で始める)

## [lang/go] 命名

- ctx / cfg / tx など Go で確立された短縮語は使ってよい
- コンストラクタは New<型> (パッケージの主要型が一つなら New)
  - functional option は With<項目>
- 型変換を関数で表す場合は To<型>
- 型変換をメソッドで表す場合は目的の型名にする
- getter は動詞を付けず対象名そのものにする
- レシーバ名は型を表す 1〜2 文字の短縮にする

## [lang/go] 分岐

- `switch` には必ず `default` 節を書く
